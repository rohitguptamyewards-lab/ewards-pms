1. Executive Summary

This section is written in simple language for non-technical team members to understand what this system does and how each part works.

What is eWards PMS?

eWards PMS (Project Management System) is a web application built specifically for the eWards team. It is a single platform where the entire team can manage their daily work, track projects, log hours, report bugs, plan sprints, and much more. Think of it as your one-stop shop for everything related to work management.

Instead of using multiple tools (spreadsheets for tracking hours, emails for merchant communication, separate apps for bug tracking), everything lives in one place. Every team member logs in and sees a dashboard customized for their role.

Who Uses It?

Everyone in the eWards team uses this system, but each person sees different things based on their role:

  • CTO (Arjun) - Sees everything. Technical dashboards, cost analysis, AI tool tracking, team performance.
  • CEO (Priya) - Sees the big picture. Business metrics, feature pipeline, roadmap progress, team health.
  • Manager (Neha) - Manages day-to-day operations. Team workload, sprint planning, task assignments, reports.
  • MC Team (Amit) - Registers merchant requests. Can view request status (received, rejected, clarification needed, etc.), respond to clarifications via comments, and track linked project progress.
  • Sales (Kavitha) - Registers merchant requests. Can view request status, respond to clarification requests via comments, and track linked project progress.
  • Developer (Rohit) - Does coding work. Logs daily hours, updates task status, writes journal entries.
  • Tester (Sneha) - Tests features. Logs bugs, tracks SLA, verifies fixes.
  • Analyst (Deepak) - Analyzes data. Writes specs, reviews features, does research.

How It All Connects

Here's the big picture of how data flows through the system:

1. Ideas & Requests Come In - Merchants ask for things (Requests), team members suggest ideas (Ideas)
↓
2. Triage & Planning - MC Team/Managers review requests, good ideas get promoted to Features
↓
3. Features & Initiatives - Features are grouped under Initiatives (big goals). Priorities and deadlines are set.
↓
4. Sprint Planning - Features are added to Sprints (1-2 week work cycles) based on team Capacity
↓
5. Daily Work - Developers, Testers, Analysts work on features. They log Work Logs, Activity Logs, and write Journal entries
↓
6. Quality & Tracking - Bugs get tracked with SLA deadlines. Blockers are logged. Decisions are documented.
↓
7. Release - Completed features are deployed. Release notes are created. Changelogs are published.
↓
8. Analysis - CEO Dashboard shows business metrics. Cost Dashboard shows spending. Reports show team performance.

Architecture Overview

The application follows a monolithic architecture with the following layers:

  • Controllers - Receive requests from the browser, call services, return responses
  • Services - Business logic layer. Calculations, validations, data processing
  • Repositories - Database access layer. All SQL queries live here
  • Models - Define database table structures and relationships
  • Enums - Define allowed values for fields like status, priority, role
  • Vue Pages - Frontend components that render the UI
  • Layouts - Shared page structure (sidebar, header) used by all pages

File Structure:

  • app/Http/Controllers/Api/V1/ - All controllers (one per module)
  • app/Services/ - Business logic services
  • app/Repositories/ - Database query repositories
  • app/Models/ - Eloquent models
  • app/Enums/ - All enum definitions
  • resources/js/Pages/ - Vue page components (81 total)
  • resources/js/Layouts/ - Layout components (sidebar, header)
  • database/migrations/ - Database table definitions
  • database/seeders/ - Test data seeding
  • routes/web.php - Web routes (182 routes)
  • routes/api.php - API routes (106 routes)

6e. Requests (Merchant Requests)

What Is It?

When a merchant (customer) needs something - a new feature, a bug fix, or an improvement - the Sales or MC team submits it as a Request. The request then goes through triage to decide what action to take.

Request Flow

Received - Sales or MC Team member submits the request
↓
Under Review - Manager, CEO, or CTO is evaluating it
↓
Clarification Needed - Manager requests more information; an email is sent to the requester, who responds via comments
↓
Linked - Request is linked to an existing or new feature (accepted)

OR:

Deferred - Will be considered later
Rejected - Will not be implemented
Fulfilled - The requested feature has been delivered

Role Restrictions

MC Team & Sales roles can only:
- Submit new requests
- View request status (received, under review, clarification needed, linked, deferred, rejected, fulfilled)
- Respond to clarification requests via comments (they receive email notifications)
- View linked project status and request updates

Only CTO, CEO, and Manager can triage requests (accept, reject, defer, ask clarification).

Email Notifications

  • Clarification Needed - Email sent to the original requester with the reason, prompting them to add a comment
  • Request Accepted/Rejected/Deferred - Email sent to the requester with the triage decision and reason

↓
Active - Sprint is running. Team is working on committed features.
↓
Completed - Sprint is over. Review what was done vs. what was planned.
↓
Archived - Historical record.

6j. Work Logs (Clockify-Style Time Tracker)

What Is It?

Every team member logs their daily work hours here. The interface is designed like Clockify - a professional time tracking tool. You type what you worked on, select a project, set start and end time, and click ADD. Everything happens on one page - no separate "create" form needed.

Page Layout

URL: /work-logs

The page has three sections:

  1. Header Bar - Title "Work Logs", a weekly total display (e.g., "This Week 32:45:00"), and a Filters toggle button
  2. Inline Entry Bar - A form bar at the top where you type your entry and click ADD (like Clockify's top bar)
  3. Entries Grouped by Date - All your logged entries grouped under date headers (e.g., "Mon, Apr 6") with daily totals

How to Log Work (Step by Step)

  1. Go to Work Logs from the sidebar
  2. In the inline entry bar at the top of the page, type what you worked on in the "What have you worked on?" field
  3. Click the Project dropdown (searchable) and select your project. A colored dot appears next to the project name
  4. Optionally select a Task from the dropdown (only shows tasks for the selected project)
  5. Set the Status (Done, In Progress, or Blocked). If Blocked, a blocker description field appears
  6. Set the Date (defaults to today)
  7. Enter Start Time (auto-filled from your last log's end time) and End Time
  8. The Duration is calculated automatically and displayed (e.g., "02:30")
  9. Click the green ADD button to save
  10. Your entry immediately appears in the grouped list below

Entry List Features

  • Grouped by Date - Entries are organized under date headers (e.g., "Sat, Apr 5") with the number of entries and daily total hours
  • Each Entry Shows: Description, project name with color dot, task name, status badge, time range, duration in HH:MM format
  • Hover Actions: When you hover over an entry, three action buttons appear:
    • Edit (pencil icon) - Edit the note, times, and status inline
    • Delete (trash icon) - Delete the entry (with confirmation)

  • Weekly Total - Displayed prominently in the header showing your total hours for the current week (Mon-Sun)

6z. Cost Rates & Cost Dashboard

What Is It?

The cost system calculates how much each feature costs to build based on team members' hourly rates and the hours they spend. Only the CTO has access.

How Cost Calculation Works

  1. Cost Rates are set for each team member: monthly CTC (salary), working hours per month, and an overhead multiplier (default 1.3x for benefits, office costs, etc.)
  2. Loaded Hourly Rate = (Monthly CTC * Overhead Multiplier) / Working Hours per Month
  3. When someone logs activity on a feature, the system multiplies their hours by their hourly rate
  4. Feature Cost = Sum of (each person's hours * their hourly rate)

9. Application Flows (Overview)

Flow 1: Merchant Request to Feature Release

Step 1: Merchant contacts Sales team about a need
↓
Step 2: Sales person logs into PMS, goes to Requests > Create, fills in details (title, type, urgency, merchant, revenue impact)
↓
Step 3: Request appears on MC Team's dashboard as "Untriaged"
↓
Step 4: MC Team reviews the request, checks for duplicates, and triages it: Accept (link to feature), Defer, or Reject
↓
Step 5: If accepted, a Feature is created (or linked to existing feature). Feature starts in "Backlog" status
↓
Step 6: During Sprint Planning, the feature is added to a Sprint with committed hours
↓
Step 7: Developer works on the feature. Logs Work Logs and Activity Logs daily. Updates feature status: In Progress > In Review
↓
Step 8: Tester tests the feature. Status moves: Ready for QA > In QA > Ready for Release
↓
Step 9: Feature is included in a Release. Deployed to staging, then production.
↓
Step 10: Changelog is written, approved, and published. Request status becomes "Fulfilled".

Flow 2: Daily Team Member Workflow

Morning: Log in > See Dashboard > Check assigned tasks and blocked items
↓
Start Work: Go to Work Logs > Log Work. Start time auto-fills from last log's end time
↓
During Day: Work on tasks. If blocked, create a Blocker. Update task status as needed.
↓
End of Each Work Block: Complete the work log entry. Start a new one for the next block.
↓
End of Day: Fill in Activity Log (more detailed). Write Journal entry (accomplishments, blockers, mood, plans).

Flow 3: Sprint Cycle

Planning (Day 1): Manager creates a new Sprint. Sets goal, dates, and total capacity hours.
↓
Feature Selection: Add features to the sprint with committed hours per feature. Check capacity.
↓
Activation: Manager activates the sprint. Status: Planning > Active.
↓
Execution (1-2 weeks): Team works on committed features. Daily logging and updates.
↓
Completion: Sprint ends. Manager marks it Complete. Unfinished features are carried over with reasons.

Flow 4: Bug SLA Tracking

Bug Found: Tester or user finds a bug. Create Bug SLA record with severity (P0-P3).
↓
SLA Clock Starts: System auto-calculates deadline (P0=4h, P1=24h, P2=72h, P3=next sprint).
↓
Fix & Close: If fixed before deadline, SLA is met. If not, "breached_at" is recorded.
↓
Reopen: If the bug comes back, the reopen count increases and a new SLA cycle starts.

10j. Complete Request Lifecycle (End-to-End with All Stages)

This is the most detailed view of how a merchant request flows through the entire system, from initial contact to delivery:

Stage 1: RECEIVED
Who: MC Team or Sales submits the request
What happens: Request is created in the system. Status = "Received". It appears as "Untriaged" on the Manager Dashboard.
Email: None at this stage
↓
Stage 2: UNDER REVIEW
Who: CTO, CEO, or Manager opens the request
What happens: The triage process begins. Manager checks for duplicates, evaluates urgency and revenue impact.
↓
Stage 2a: CLARIFICATION NEEDED (optional)
Who: Manager needs more information
What happens: Manager clicks "Ask Clarification" and types a question. Status changes to "Clarification Needed".
Email: Sent to the original requester (MC Team/Sales member) with the question.
Response: Requester logs in, reads the question in comments, types a reply. Flow returns to Stage 2.
↓
Stage 3a: LINKED (Accepted)
Who: Manager accepts the request
What happens: A new Feature is created (or request is linked to an existing feature). Request status = "Linked". Feature status starts at "Backlog".
Email: Sent to requester: "Your request has been accepted and linked to feature [name]"
↓
Stage 4: FEATURE IN DEVELOPMENT
Who: Development team
What happens: Feature moves through pipeline: Backlog → In Progress → In Review → Ready for QA → In QA → Ready for Release
↓
Stage 5: RELEASED
Who: Manager creates a Release
What happens: Feature is included in a Release. Deployed to production. Feature status = "Released".
Changelog: Written, approved by manager, published.
↓
Stage 6: FULFILLED
Who: Automatic
What happens: When the linked feature is released, the request status automatically changes to "Fulfilled". The merchant's need has been delivered.

Alternative Outcomes (instead of Stage 3a):

DEFERRED: Manager decides to postpone. Request can be re-evaluated later. Email sent to requester.
REJECTED: Manager decides not to implement. Reason is recorded. Email sent to requester with rejection reason.

Key Technical Patterns

  • Soft Deletes: Records are never truly deleted. A deleted_at timestamp is set instead. This allows recovery.
  • Polymorphic Relationships: Comments and Documents can be attached to any entity type (project, task, feature, etc.) using commentable_type/commentable_id columns.
  • Enums: All status fields, types, and categories use PHP enums for type safety. There are 50 enum definitions.
  • Audit Trail: The Auditable trait automatically logs all create/update/delete actions to the audit_logs table.
  • Role-based Access: Controllers check the user's role to determine what data to show and what actions to allow.
  • Encrypted Fields: Git provider credentials and webhook secrets are encrypted at rest using Laravel's encryption.
  • Pagination: All list endpoints use Laravel's pagination (default 15 per page) for performance.
